## 一、项目是什么

**NetWatch（网络体检）** — macOS 原生网络监控 + 代理安全检测 + ChatGPT/Claude 封禁风险评估。

一个 24 小时挂在后台的 macOS 工具，每分钟给网络体检一次。能分层定位网络故障（路由器→国内→代理软件→代理通道→国外），还能检测代理 IP 的纯净度评分、被风控概率、代理位置历史。App 内一键更新（从 GitHub 拉代码 → 本地编译 → 替换重启）。

**GitHub**: https://github.com/julyyy-666/NetWatch

## 四、核心逻辑详解

### 4.1 netwatch.sh — 网络分层探测
每 60 秒执行，分层决策树：
```
① 路由器能 ping 通? ──否──→ 🔴 本地网络断开
    │是
② 国内(百度)能打开? ──否──→ 🔴 宽带/ISP故障
    │是
③ Surge 进程在跑? ──否──→ 🔴 代理软件未运行
    │是
④ HTTP/SOCKS 端口在听? ──否──→ 🔴 代理端口异常
    │是
⑤ 经代理能上 Google? ──否──→ 🔴 代理隧道异常
    │是
⑥ 延迟 < 3秒? ──否──→ 🟡 国外缓慢
    │是
    └──────→ 🟢 正常
```

**探测目标**（可在脚本顶部修改）：
- Surge HTTP: `127.0.0.1:6152`
- Surge SOCKS: `127.0.0.1:6153`
- 国内: `https://www.baidu.com`
- 国外: `https://www.gstatic.com/generate_204`

**输出文件**：
- `状态.json` — 最新一条采样（SwiftUI 读这个）
- `samples-YYYY-MM-DD.jsonl` — 每分钟追加
- `events.jsonl` — 仅状态变化时追加
- `当前状态.txt` — 人读快照

### 4.2 risk_check.sh — 安全评估引擎
每 600 秒执行，5 个数据源交叉验证：

| 源 | API | 免费 | 检测内容 |
|----|-----|------|---------|
| ip-api.com | `http://ip-api.com/json/{IP}` | ✅ | 基础 GeoIP + hosting/proxy 标签 |
| ipapi.is | `https://api.ipapi.is/?q={IP}` | ✅ | 精确 DC 检测 + 公司类型 + 滥用评分 |
| Shodan | `https://internetdb.shodan.io/{IP}` | ✅ | 开放端口/CPE/CVE/标签 |
| StopForumSpam | `https://api.stopforumspam.org/api?json&ip={IP}` | ✅ | 黑名单 |
| GreyNoise | `https://api.greynoise.io/v3/community/{IP}` | ✅ | 已知安全/恶意 |

**评分模型**（100 分递减，学自 ByeByeVPN）：
- 数据中心 IP → -30 分
- VPN 标记(≥2源) → -18 分
- Tor 出口 → -25 分
- 黑名单 → -15 分
- VPN 端口暴露 → -15 分
- API 返回 403/451 → -30 分

**代理位置历史**：每次检测追加一条到 `代理位置历史.jsonl`，记录时间/IP/地点/评分/是否变更。频繁换地点 → 风控警告。

**API 可达性测试**：
- OpenAI: `https://api.openai.com/v1/models` → 401=可访问, 403=被封
- Anthropic: `https://api.anthropic.com/v1/messages` → 405=可访问
- Google: `https://generativelanguage.googleapis.com/` → 404=可访问

### 4.3 proxy_detect.sh — 代理软件识别
扫描进程 + 端口，识别：
- Surge (6152/6153)
- Shadowrocket (1080/1081)
- Clash Verge (7890/7891)
- ClashX (7890/7891)
- V2RayX (1087)
- sing-box (2080)
- Tailscale（额外检测）

读取 `scutil --proxy` 判断系统代理设置。

### 4.4 main.swift — SwiftUI 应用
**架构**：单 `Monitor` ObservableObject + 定时器
- `refresh()` — 每 3s 读取 JSON 文件更新 UI
- `checkForUpdates()` — 每 1h 查 GitHub Release
- `loadRiskData()` — 手动触发重新加载风险评估

**UI 布局**（从上到下）：
1. 更新横幅（有新版时显示）
2. 网络断开横幅（红色警告）
3. 状态大圆圈（🟢/🟡/🔴）
4. 路由器/国内网站检查
5. 代理软件 Surge 状态
6. 延迟柱状图（代理 vs 直连）
7. 稳定性时间线（96 格绿红条）
8. 网络质量（丢包/卡顿/延迟）
9. **账号安全体检卡片**（核心功能）
   - IP 纯净度评分 + 评分环
   - 被风控概率
   - 当前代理地点
   - 位置变更提醒
   - 代理位置历史（时间线）
   - 代理软件列表
10. 最近事件记录
11. 复制给 AI 分析按钮
12. 版本号 + 更新检查状态

**自动更新流程**（UpdateBanner.autoUpdate）：
```
1. 从 GitHub raw 下载 main.swift
2. 本地 swiftc 编译
3. 写 update_helper.sh 到 ~/Library/Application Support/NetWatch/
4. nohup 启动 helper（独立进程）
5. NSApp.terminate(nil) 退出 app
6. helper: sleep 2 → rm -rf 旧.app → 重建 .app → 签名 → open 重启
```

## 十、设计决策记录

1. **为什么用 Bash + SwiftUI 混合架构**：Bash 脚本零依赖、launchd 可靠、不占资源（每分钟跑几秒）。SwiftUI 只负责读 JSON 显示。解耦了探测和展示。
2. **为什么不用 Swift 原生网络探测**：Bash 的 curl/ping/nc 在 macOS 上最稳定，不受 App Sandbox 限制。Swift 进程可以读文件但网络请求可能被沙箱拦。
3. **为什么用文件而非进程间通信**：JSON 文件是最简单的 IPC，调试方便（直接 cat 看），不怕 crash 丢数据。
4. **为什么版本号从 Info.plist 动态读取**：之前写死 `currentVersion = "4.0"` 导致永远显示有更新。改成 `Bundle.main.infoDictionary` 后，app 编译时是什么版本就是什么版本。
5. **为什么自动更新要删整个 .app 重建**：macOS 签名保护机制不允许 `cp` 覆盖正在运行的 .app 内文件（`Operation not permitted`）。必须先 `rm -rf` 整个 .app 再重建。
6. **为什么用 nohup 启动 helper**：app 退出后子进程也会被杀。`nohup bash script &` 让 helper 脱离父进程，app 死了 helper 活着。
